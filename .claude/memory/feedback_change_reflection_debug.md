`scripts/ui.html` や `scripts/server.py` を編集したのにブラウザに反映されない時、
**ロジックを疑う前にこの5段階を順に確認する**。今回のセッションで複合的に踏みまくった罠。

## チェックリスト（上から順に）

### 1. ファイル保存確認
`grep -c "<追加したマーカー>" scripts/ui.html` が >0 か。
エディタが書き込んでいない / 別のパスを編集した、を排除。

### 2. サーバープロセスが「ちょうど1個」listen しているか
```bash
netstat -ano | grep ":8000.*LISTENING"
```
- **2つ出てきたら危険**。古い + 新しいサーバーが両方 bind していて、リクエストがランダムに振られる
  → 古い方は起動時にキャッシュした旧 ui.html を返し続け、新挙動が出たり出なかったり
- `taskkill /IM python.exe` は他用途で失敗することがあるので、**PID 指定で個別 kill** が確実

### 3. サーバーが新しい内容を返しているか
```bash
curl -s http://localhost:8000/ | grep -c "<追加したマーカー>"
```
これが 0 ならサーバー側で読み直しできていない（INDEX_HTML 等の起動時キャッシュ復活を疑う）。
**server.py の `do_GET` は ui.html をリクエスト都度 read_text で読み直すべし**（既にそうなっている。
将来キャッシュ最適化したくなっても戻さない）。

### 4. ブラウザがキャッシュ HTML を掴んでいないか
**Ctrl+Shift+R**（ハードリロード）。普通のリロード（F5）では JS/CSS が更新されないことがある。

### 5. localStorage に古い値が残って hardcode を上書きしていないか
特に「デフォルト値の意味を変えた」時はこれが罠。`LSKEY` のバージョンバンプで強制リセット。
[[feedback-localstorage-version]] 参照。

## 1-5 すべてクリアして初めてロジックバグを疑う

**Why:**
ありがちな失敗連鎖：
- 「変えたのに動かない」→ 焦ってロジックを更に書き換える → 余計におかしくなる
- 実際は #2（古いサーバー残存）or #5（localStorage）が原因なのに、コードを延々と直していた
- 今回のセッションで実際に 30分+ 溶かした

## How to apply

ユーザーが「動かない / 反映されない」と言ったら、まずこの順で潰す。
ロジック修正パッチを書き始めるのは #1-#5 すべて OK になってから。
